## 2. How the maintainer works (this is most of the job)

- He iterates in rounds of annotated screenshots. Colour code: RED = remove /
  wrong, BLUE = restore / keep-this, GREEN = keep, arrows = "should extend
  here". Sometimes a single blue dot = a coordinate (e.g. a rotation pivot).
- Apply marks LITERALLY. When he says "apply my marks literally", he means
  pixel-for-pixel — register his screenshot to the asset (see §4) and apply
  each marked cell. Don't improvise beyond the marks; don't "improve" nearby
  pixels he didn't mark. When I tried to be clever (block-snapping a mask to
  "make it look like pixel art again"), it was rejected flat: "The before
  looks much better, so keep the before."
- His reposted images accumulate JPEG artifacts ("Each time I repost an image
  the graphics get more and more jpeg artifacts") — prefer the FIRST upload
  of any asset as pixel source, and keep your own extracted intermediates.
- AI-regenerated reference images are NOT pixel ground truth: "It has been
  copied and regenerated by AI... pixels and shapes will not align
  correctly." Use them for intent and for borrowed FILL pixels, registered
  locally (per-band, small offsets), never as a global overlay.
- Always send verification crops back (SendUserFile) — he reviews from his
  phone. Crop tight to what changed, downscale big screenshots.

## 3. Golden rules (violations get caught immediately)

- PIXEL ART SCALES NEAREST-NEIGHBOUR ONLY. Everywhere. `image-rendering:
  pixelated` on every img/canvas, `imageSmoothingEnabled=false` in canvas
  code, NEAREST in preview scripts. Box-averaging is allowed ONLY when baking
  an asset DOWN to its final display resolution.
- Uploads are blurry upscales. ALWAYS find the native pixel grid first
  (measure block size, or register against a known-native sibling asset),
  box-downscale to native, THEN key/extract. Keying at upload resolution
  gives ragged edges — this failed once ("Your way of extracting the
  graphics makes it look terrible!").
- Keying: use hue-axis channels, not distance-to-colour alone. Magenta bg:
  alpha from `min(r,b)-g`, then unmix `art=(px-(1-a)*MAG)/a`. Teal bg:
  beware DARK pixels and DARK GREEN — both look "teal" to naive tests
  (`g-r` alone eats vines; add `b > g-25`-style guards). Decontaminate edge
  pixels by masked averaging (average art pixels only, then downscale).
- Always drop floating components (largest-connected-component keep) and
  check the result on BOTH white and dark backdrops — halos hide on dark.
- Buttons/slots keep a crisp 1px black border by construction (paint the
  mask boundary), soft alpha elsewhere; a rotating sprite (the clock hand)
  keeps SOFT averaged alpha or rotation shreds it.
- Never hand-fix individual pixels when re-extraction is possible: "I want
  you to reextract the image and not try to fix individual pixels."

## 4. Registering his annotated screenshots

The recipe that works every round: detect the saturated mark colours
(`r>190,g<95,b<95` etc.), mask them out, then grid-search scale (0.05 steps)
× integer offsets minimizing masked L1 against the target asset/frame. His
phone screenshots land anywhere from 5.6× to 11.6× — never assume. Residual
~3/channel = good registration; ~10+ = wrong scale or a JPEG re-post.
Then box-downscale the mark masks to asset cells (threshold ~45% coverage)
and apply red/blue per cell.

## 5. The frame system (frame2.ts) — read the file header, it's all there

One runtime-composed canvas from `/ui2/frame.png` (768×1376, the
pixel-perfect extraction — 15 rounds, don't regress it) + rune-free top aux.
Stretching inserts pixels at maintainer-approved cut lines; every stretch
repeats PLAIN texture only (single-column/row extrusions of the exact cut
column — joints must be originally-adjacent pixels); decorated art (runes,
crystals, corners, clock disc) only ever lives in fixed sections.
`splitInsH` currently sends ALL vertical head-room to the game area
(maintainer wants max game view; the page stretch is wired but unused).
Interior windows are MEASURED off frame.png's alpha (rail A opaque band
665–693 → split 676; page window x48..720 y874..1306; tab band same x).
If you touch frame.png, patch `frame-top-runefree.png` rows 0–99
identically — they must stay twins. FrameLayout is the contract consumed by
hud.ts (game split, chat anchor, tab/page rects, clockAnchor) — changing its
semantics touches MY side (game canvas height, chat anchors); ping me.

## 6. Layout glue and the uiZoom trap

Two coordinate spaces coexist: the FRAME (and its canvas, and now the clock
HAND) are plain layout px, never uiZoom'd; overlays (clock dials, badge,
chat, select) carry a compensating CSS `zoom` (uiscale.ts) because of the
"Desktop site" neutralization. A px anchor inside a zoomed overlay renders
at value×zoom — anchoring an overlay to a frame feature needs
`calc(px / var(--ml-uizoom,1))`. This bit us twice. z-stack (bottom→top):
game canvas → overlay cover (3) → .ml-hud (4) → clock dials (5) → frame
canvas (6) → clock hand (7). Chat log 5 / input 6.
